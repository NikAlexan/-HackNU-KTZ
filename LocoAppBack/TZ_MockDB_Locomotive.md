# Техническое задание: Генерация моковой базы данных телеметрии

**Хакатон-кейс:** Визуальный цифровой двойник локомотива  
**Среда:** Claude Code · **Версия:** 1.0 · **Дата:** Апрель 2026

---

## 1. Цель и контекст

Скрипт генерирует реалистичную моковую БД, покрывающую два класса тяги — **электровозы (25 кВ AC)** и **тепловозы (дизель)**. База служит источником данных для:

- Дашборда машиниста — real-time телеметрия, индекс здоровья, алерты
- Дашборда диспетчера — мониторинг смешанного флота, события, рекуперация
- Backend API — SSE/WebSocket-эндпоинты с имитацией потоковой передачи
- Нагрузочного тестирования — highload-сценарии (x10 burst)

> **Важно:** Данные должны быть физически правдоподобны: отклонения внутри реальных диапазонов, коррелированные параметры (рост тока → рост температуры), нормальные и аварийные сценарии.

## 5. Сценарии данных

Каждый локомотив должен иметь **минимум 1 нормальный и 1 аномальный** сценарий за период генерации.

### 5.1 Нормальный рейс (`NORMAL_RUN`)
- Скорость: плавный разгон → крейсерская → торможение перед остановкой
- Температуры: медленный рост при нагрузке, стабилизация на крейсере
- Электровозы: рекуперация активна при торможениях (`traction_mode = REGEN`)
- Тепловозы: расход топлива коррелирует с оборотами и нагрузкой

### 5.2 Перегрев (`OVERHEAT`)
- Температура масла / трансформатора монотонно растёт, пересекает порог 90°C
- Появляется `error_code` (`E011` — перегрев масла, `E041` — перегрев трансформатора)
- Скорость снижается (машинист реагирует) — задержка 30–60 сек
- `healthIndex` падает: 95 → 85 → 74 (grade: A → B → C)

### 5.3 Просадка напряжения (`VOLTAGE_SAG`) — только `ELECTRIC`
- `catenary_voltage_kv` падает с 25.0 до 22.5–23.5 кВ за 2–3 минуты
- `pantograph_current_a` растёт обратно пропорционально (P = U × I)
- `power_factor` снижается до 0.78–0.82
- `error_code: E021` — low catenary voltage

### 5.4 Критический алерт (`CRITICAL_ALERT`)
- Один из параметров резко выходит за критичный порог
- `healthIndex` падает ниже 65 (grade: D или E)
- Генерируются 2–3 связанных события в `events.json`

### 5.5 Highload burst (`BURST`)
- Отдельный массив `burst_readings` — 10x плотность (`interval_ms: 50`)
- Длительность: 30 секунд, имитирует пиковую нагрузку
- Флаг `is_burst: true` в каждом reading
- Используется для тестирования SSE/WebSocket устойчивости фронтенда

> **Важно:** Backend должен уметь фильтровать или агрегировать burst-данные при необходимости.

## 6. Алгоритм расчёта Health Index

`healthIndex = 100 − Σ(penalty_i × weight_i)`, где `penalty_i` — штраф за отклонение параметра.

### 6.1 Веса и пороги

| Параметр                              | Порог предупреждения | Порог критичного | Вес   | Макс. штраф |
|---------------------------------------|----------------------|------------------|-------|-------------|
| Температура масла / трансформатора    | > 80°C               | > 95°C           | 0.25  | 25          |
| Напряжение сети (только ELECTRIC)     | < 23 кВ              | < 20 кВ          | 0.20  | 20          |
| Ток ТД (макс. из всех)               | > 550 А              | > 650 А          | 0.20  | 20          |
| Давление тормозов                     | < 4.5 / > 6.0        | < 3.5 / > 6.5    | 0.20  | 20          |
| Уровень топлива (только DIESEL)       | < 40%                | < 20%            | 0.10  | 10          |
| Наличие `error_code`                  | любой                | E0x1+ (крит.)    | 0.05  | 5           |

### 6.2 Буквенные оценки

| Диапазон | Оценка | Статус                            |
|----------|--------|-----------------------------------|
| 90–100   | **A**  | Отлично — всё в норме             |
| 75–89    | **B**  | Хорошо — незначительные отклонения |
| 60–74    | **C**  | Удовлетворительно — требует внимания |
| 40–59    | **D**  | Плохо — требует вмешательства     |
| 0–39     | **E**  | Критично — немедленные действия   |

Функция экспортируется как:

```typescript
export function calculateHealthIndex(
  reading: TelemetryReading,
  locoType: 'ELECTRIC' | 'DIESEL'
): { index: number; grade: 'A' | 'B' | 'C' | 'D' | 'E'; penalties: PenaltyDetail[] }
```

## 10. Технические требования

### 10.1 Стек и зависимости

| Пункт     | Требование                                                        |
|-----------|-------------------------------------------------------------------|
| Runtime   | Node.js 20+ / Bun                                                 |
| Язык      | TypeScript, `strict: true`                                        |

### 10.2 CLI-аргументы `seed.ts`

| Аргумент          | По умолчанию | Описание                                              |
|-------------------|--------------|-------------------------------------------------------|
| `--locos N`       | `12`         | Общее количество локомотивов в флоте                  |
| `--hours H`       | `1`          | Период генерации телеметрии в часах                   |
| `--interval-ms N` | `500`        | Интервал между записями телеметрии                    |

| `--format`        | `json`       | Формат вывода: `json \| sqlite \| csv`                |

### 10.3 Опциональный вывод в SQLite (`--format sqlite`)

Создаёт файл `mock.db` с таблицами:
- `locomotives` — один ряд на локомотив
- `telemetry_readings` — партиционирована по `loco_id`
- `events` — лог событий
- `routes` и `route_waypoints`

Индексы обязательны: `telemetry_readings(loco_id, ts)` для быстрой выборки последних N записей.

### 10.4 Производительность

- 1 час × 12 локомотивов × 2 события/сек = ~86 400 записей — генерация **< 10 секунд**
- Использовать потоковую запись (`fs.createWriteStream`) — не держать всё в памяти
- Прогресс-бар в консоли обязателен (`ora` или ручной `\r`)

## 11. Скелет кода

```typescript
// generators/telemetry.ts
export function generateTelemetryStream(
  loco: Locomotive,
  opts: { hours: number; intervalMs: number; scenario: ScenarioType }
): TelemetryReading[] {
  const readings: TelemetryReading[] = [];
  const totalMs = opts.hours * 3600 * 1000;
  let state = initState(loco);         // начальное состояние

  for (let t = 0; t < totalMs; t += opts.intervalMs) {
    state = evolveState(state, loco, opts.scenario, t);
    readings.push(stateToReading(state, new Date(BASE_TS + t)));
  }
  return readings;
}
```

Функция `evolveState` применяет физические модели:
- ускорение / торможение
- теплообмен: `dT/dt ∝ нагрузка − охлаждение`
- изменение напряжения контактной сети
- расход топлива коррелирует с оборотами дизеля
